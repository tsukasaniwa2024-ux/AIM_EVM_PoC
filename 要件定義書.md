# 要件定義書

**プロジェクト名**：EVM 輸入帳票自動化システム  
**作成日**：2026-06-24  
**バージョン**：1.2  

> **本ドキュメントはPoC（概念実証）フェーズの要件定義書です。**  

---

## 1. 背景・目的

### 1.1 背景

AIM株式会社では中国から日本への輸入業務において、経理処理に使用する帳票（PDF・画像）から必要なデータを手作業で転記・計算している。この作業は工数がかかり、転記ミスのリスクも存在する。

### 1.2 目的

- 帳票データの読み取りをAI（OpenAI Vision API）で自動化し、手作業による転記を廃止する
- 計算処理を自動化して事業企画部担当者の工数を削減する
- 処理結果をCSV / Excelで出力し、既存の経理フローに接続できるようにする
- データをDBに蓄積して将来的な分析・監査対応に備える

---

## 2. スコープ

### 2.1 対象

- 中国→日本 輸入に関する経理帳票（PDF形式 および 画像形式）
- 処理結果の出力（CSV / Excel）
- 処理履歴のDB保存

### 2.2 対象外

- 日本→中国 輸出帳票
- 会計システムへの直接連携（将来拡張）
- ユーザー認証・権限管理（将来拡張）
- 送料・関税の自動計算（将来拡張）

## 3. 利用者

| ロール | 説明 |
|--------|------|

## 4. 機能要件

### 4.1 ファイルアップロード・入力

| ID     | 要件 |
|--------|------|
| F-01 | ユーザーはブラウザ上でPDFファイルを1件選択できる |
| F-02 | ユーザーはブラウザ上で画像ファイル（PNG / JPG）を1件選択できる |
| F-03 | PDFと画像の両方が選択されていない場合、処理を実行できない |
| F-04 | ファイルサイズ・ページ数の上限は設けない（目安として1ファイル10ページ以内を推奨）。PoCフェーズでの制限は実運用時に見直す |
| F-05 | 処理実行前に為替レート（円換算値）を手動入力できるフォームをUIに表示する。初期値は空欄とする |
| F-06 | 為替レートの入力範囲は1〜999とする。範囲外の値が入力された場合はエラーとして処理を実行できない |
| F-07 | 為替レートの決定優先順位は以下の通りとする。①PDFから抽出された値、②画像から抽出された値、③UIへの手動入力値。①②③すべて存在しない場合はエラーとして処理を実行できない |

### 4.2 OCRデータ抽出

| ID     | 要件 |
|--------|------|
| F-08 | PDFファイルをOpenAI Vision APIに送信し、帳票データをJSON形式で抽出できる |
| F-09 | 画像ファイルをOpenAI Vision APIに送信し、帳票データをJSON形式で抽出できる |
| F-10 | 抽出対象項目は帳票の種類・フォーマットに依存するため固定しない。AIが読み取れたすべての項目を抽出対象とする |
| F-11 | 抽出できなかった項目は「抽出できず」として画面上に表示する |
| F-12 | OCRにはOpenAI Vision APIを使用する。将来的な差し替えに備え、使用するAIプロバイダは`.env`の設定変更のみで切り替え可能な設計とする |

### 4.3 計算処理

| ID     | 要件 |
|--------|------|
| F-13 | 品目明細の`amount`フィールドに為替レートを掛けて円換算額（`amount(円換算)`）を算出し、品目明細の末尾列に表示する |
| F-14 | `amount`フィールドが存在しない品目の円換算額は空欄とする |

### 4.4 結果プレビュー

| ID     | 要件 |
|--------|------|
| F-15 | 「実行」ボタン押下後、AI解析結果を画面上に表示する。AI解析が失敗した場合はエラーメッセージを表示し、ファイル出力ボタンを非活性にする |
| F-16 | プレビューは「基礎情報」と「品目明細」の2セクションに分けて表示する |
| F-17 | 基礎情報は「項目名・値・抽出元」の表形式で表示する |
| F-18 | 品目明細は帳票から抽出されたキーを動的にカラムとして表示する（固定カラムなし） |

### 4.5 ファイル出力

| ID     | 要件 |
|--------|------|
| F-19 | 「CSVで出力」ボタン押下でCSVファイルをダウンロードできる。保存場所は利用者が任意で選択できる |
| F-20 | 「Excelで出力」ボタン押下でExcel（.xlsx）ファイルをダウンロードできる。保存場所は利用者が任意で選択できる |
| F-21 | 出力ファイルには基礎情報・品目明細の両方を含める（Excelは別シート、CSVは別セクション） |
| F-22 | 出力ファイルのファイル名には処理日時を含める（例：`import_20260624_153000.csv`） |

### 4.6 DB保存

| ID     | 要件 |
|--------|------|
| F-23 | 処理ごとに基礎情報・処理日時をDBに保存する |
| F-24 | ローカル・イントラ環境ともにSQLiteを使用する |
| F-25 | DB接続先は`DATABASE_URL`環境変数で管理する |

## 5. 非機能要件

### 5.1 性能

| ID     | 要件 |
|--------|------|
| NF-01 | 1件の処理（OCR＋計算＋DB保存）を30秒以内に完了する |
| NF-02 | 同時利用者は1〜3名を想定する（ローカル環境） |

### 5.2 可用性・保守性

| ID     | 要件 |
|--------|------|
| NF-03 | Dockerで環境を再現できること（`docker-compose up --build`で起動） |
| NF-04 | OCRプロバイダを`.env`の`OCR_PROVIDER`設定変更のみで差し替えできること |
| NF-05 | 環境依存の設定値はすべて`.env`で管理すること（APIキー、DB URL等） |

### 5.3 セキュリティ

| ID     | 要件 |
|--------|------|
| NF-06 | APIキーをソースコードにハードコードしない |
| NF-07 | アップロードファイルはPDF / PNG / JPGのみ受け付けるバリデーションを行う |

### 5.4 拡張性

| ID     | 要件 |
|--------|------|
| NF-08 | 計算ロジックを`calculator.py`に集約し、ロジック変更がバックエンド全体に影響しない設計とする |
| NF-09 | DBのテーブル設計はフィールド追加が容易なEAV構造（key-value形式）とする |

## 7. 用語定義

| 用語 | 説明 |
|------|------|
| 帳票 | 輸入業務で使用するインボイス・パッキングリスト等の書類 |
| PoC | Proof of Concept（概念実証）。機能の実現可能性を確認するための試作フェーズ |
| OCR | 画像・PDFからテキストを抽出する技術。本システムではOpenAI Vision APIを使用 |
| 為替レート | 外貨→日本円の換算レート。帳票上の値またはUIへの手動入力値を使用（範囲：1〜999） |
| 基礎情報 | 帳票から抽出した日付・番号・取引先・合計金額等のヘッダー情報 |
| 品目明細 | 帳票から抽出した品番・品名・数量・単価・金額等の明細行情報 |
| EVM | AIM株式会社の社内プロジェクト名称 |

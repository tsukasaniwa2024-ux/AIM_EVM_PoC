## テスト対象範囲

- フロントエンド：PDF1件＋画像（JPG/PNG）1件の選択、為替レート取得方法のラジオボタン選択、OCR解析・計算実行、結果プレビュー表示、CSV/Excel出力

- DB：`import_sessions` / `import_fields`（EAV構造）への保存内容

## 判定基準（凡例）

| 判定 | 意味 |
|---|---|
| OK | 期待結果どおりに動作した |
| NG | 期待結果と異なる／不正な動作をした（バグ票を発行し備考に番号を記載） |
| 未実施 | まだテストを実行していない |
| 対象外 | 今回のPoCスコープでは確認しない |

## 補足：実装側への確認推奨事項

1. **自動取得モード時のマージ戦略適用範囲**：「帳票から自動取得」選択時のみ`merge_strategy`（pdf_first等）が適用される想定だが、この前提で問題ないか実装側に確認する。
2. **手動入力モード時の帳票内レートの扱い**：手動入力モードでは帳票内に記載があっても完全に無視する仕様で問題ないか確認する（TC-029）。
3. **モード切替時の入力欄クリア有無**：自動↔手動を往復した際、手動入力欄の値が保持されるかクリアされるか、UI仕様として明確化しておくと開発・テストの手戻りを防げる。
4. **要件定義書 F-07/F-11／用語定義 では OCRに「OpenAI Vision API」を使用すると明記**されていますが、基本設計書のOCR抽象化レイヤーには Google Vision／AWS Textract の実装例のみが示されており、OpenAI Vision用の実装クラスが明示されていません。テスト実施前に開発チームへ実装方針を確認してください。
